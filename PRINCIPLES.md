# Design Principles

## 1. Evidence First

Evidence always wins over opinion.

Every statement must be backed by evidence.

No unsupported claims.

---

## 2. Transparency Over Certainty

Confidence is earned.

Every claim should clearly indicate:

- Unknown
- Possible
- Probable
- Confirmed
- Disproven

Never overstate certainty.

---

## 3. Explainable AI

AI assists.

Humans conclude.

Every AI-generated conclusion must be traceable back to evidence.

---

## 4. Generic Before Specific

Never build:

BibleTimeline

Build:

Timeline

Never build:

BibleOwner

Build:

Person

Never build:

BibleResearch

Build:

Investigation

Everything should be reusable.

---

## 5. Database Before Interface

The data model comes first.

The interface is simply another view of the data.

If it isn't modeled correctly,
it shouldn't exist in the UI.

---

## 6. API First

Every feature should eventually be accessible through an API.

Design systems—not pages.

---

## 7. Version Everything

Research changes.

Evidence changes.

History changes.

Nothing should ever be overwritten.

Every investigation should maintain a complete version history.

---

## 8. Images Are Evidence

Images are not decoration.

Images are primary historical sources.

Every image should support:

- Zoom
- Metadata
- Annotation
- Provenance
- Source references

---

## 9. Relationships Matter

Every object should connect to:

- People
- Places
- Events
- Artifacts
- Documents
- Organizations
- Collections

History is a graph.

Not a folder.

---

## 10. Timelines Are Dynamic

Every:

Artifact

Person

Place

Investigation

Organization

should automatically generate a timeline.

---

## 11. Documentation Is Code

Every architectural decision should be documented.

Future developers should understand WHY—not just HOW.

---

## 12. Performance Builds Trust

Fast software earns trust.

Target:

- Lighthouse 95+
- Server Components first
- Accessibility built-in
- Minimal JavaScript

---

## 13. Security by Default

Least privilege.

Encrypted secrets.

Audit logs.

No shortcuts.

---

## 14. Enterprise Ready

Every feature should assume future support for:

- Multiple organizations
- Multiple collections
- Permissions
- Roles
- Workflows
- Approvals
- APIs
- Exports
- Integrations

---

## 15. Build for Acquisition

Before writing code ask:

"Would this make Project 250 easier to acquire in 2030?"

If not—

rethink it.

# The Knowles Principle

Every great investigation begins with genuine curiosity.

Project 250 began with four forgotten English Bibles purchased from The Ancient House Bookshop in Ipswich, Suffolk, around 1990.

For decades they sat quietly on a shelf until one question was asked:

**"Who owned these books?"**

That single question uncovered:

- Forgotten families
- Victorian inscriptions
- Parish records
- Rare maps
- Literary connections
- Four centuries of provenance
- A community of historians, genealogists, collectors, and researchers

The software exists because the investigation came first.

We believe:

Every artifact has a story.

Every story deserves evidence.

Every conclusion must be traceable.

Every discovery should invite another question.

We are not building software to tell people what happened.

We are building software that allows people to understand WHY we believe it happened.

Trust is earned through transparency.

History is preserved through evidence.

Curiosity is the beginning of every Case File.
